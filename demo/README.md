# Demos

## `full-loop.sh` — the whole story in one hermetic command (start here)

```sh
demo/full-loop.sh        # no API key, no network; ~1 min including build
```

Walks the complete loop end-to-end on local fixtures and prints a labeled summary:

```text
upstream change → import → agent transform → verifier → export → replay
```

- An upstream repo (`acme/greeter`) renames `Connect()` → `New()`, leaving an
  in-package caller broken.
- `capyfun import` mirrors the history, runs an **agent transform** to migrate the
  caller, verifies with `go test`, and materializes the agent's edits to a
  content-addressed patch.
- `capyfun export` ships a monorepo path back out as a PR branch, scrubbing
  internal-only lines.
- Re-running the import is a **cache hit** — the agent does not run again (model
  calls: 0, cost: $0.00).

The agent runs through the **`fixture` executor**: a deterministic *mock* agent
(recorded edits, no model, no network), clearly labeled as such. The same edge
runs a real coding agent with `--executor local`. For the measurable eval table
across 3 fixtures, see [`scripts/eval-agents.sh`](../scripts/eval-agents.sh) and
[`docs/evals.md`](../docs/evals.md).

## `scripts/smoke-react.sh` — issue → agent → PR, hermetic

```sh
scripts/smoke-react.sh        # no network, no real agent; rerunnable
```

Exercises the **`on_issue` reaction** end-to-end: it builds a local bare origin
for `acme/backend`, stubs the agent harness with a fake `claude` that edits the
checkout, and runs the real `capyfun react` CLI over a saved `issues` webhook
payload (`testdata/issue-labeled.json`) against the
[`examples/reactions/`](../examples/reactions/) config.

It first does a `--dry-run` (resolve + match only), then the live reaction
(clone → branch `capyfun/issue-7` → agent edit → commit → push → record PR via the
hermetic `LocalForge`), and asserts the prototype branch landed with the agent's
change plus the `CapyFun-Agent` / `CapyFun-Issue` provenance trailers. Against a
real GitHub remote, the GitHub-App `Forge` clones/pushes and opens the PR. See
[`docs/design/reactions.md`](../docs/design/reactions.md).

## `run.sh` — multi-language dependencies as monorepo source

This demo shows CapyFun as a history-preserving alternative to per-language
vendoring (`go mod vendor`, `cargo vendor`, `node_modules`). Third-party
dependencies for three ecosystems are brought into the monorepo under
`third_party/{go,rust,js}/` — as real source, two different ways:

- **Go** → `github_import`: full upstream **history** (every first-parent commit).
- **Rust / JS** → `git_repository`: a **pinned snapshot** (one commit's tree).

It then shows the **other direction**: `github_export` ships a monorepo-native
path (`sdk/go/client/`) back *out* to a standalone repo as a PR.

## Layout

```text
demo/
├── go.mod / go.sum            # Go manifest    (read by `capyfun gen-go`)
├── Cargo.toml / Cargo.lock    # Rust manifest  (read by `capyfun gen-cargo`)
├── package.json / package-lock.json  # JS manifest (read by `capyfun gen-npm`)
├── main.go
├── SRC                        # root: monorepo()
├── sdk/
│   └── go/                    # a monorepo-native SDK, exported back OUT
│       ├── SRC                #   github_export -> github.com/acme/sdk-go
│       └── client/            #   the exported source (prefix stripped on export)
└── third_party/
    ├── go/                    # fine-grained, GENERATED by `capyfun gen-go`
    │   └── github.com/{google/uuid,pkg/errors,spf13/pflag}/SRC
    ├── rust/                  # fine-grained, GENERATED by `capyfun gen-cargo`
    │   ├── anyhow/SRC         #   one SRC per crate
    │   └── thiserror/SRC
    └── js/                    # coarse-grained, GENERATED by `capyfun gen-npm`
        └── SRC                #   one SRC, a target per package
```

## Two SRC styles

Both are valid; pick per directory:

- **Fine-grained** (Go, Rust here): one `SRC` per dependency; the import lands in
  that package's own directory (`into` defaults to the package).
- **Coarse-grained** (JS here): one `SRC` at the language root declaring multiple
  targets, each with an explicit `into` subdirectory so destinations don't
  overlap. See `third_party/js/SRC`.

## Run it

```sh
demo/run.sh        # needs network access to github.com
```

`gen-go` / `gen-cargo` / `gen-npm` scaffold the `third_party/{go,rust,js}` rules
from each ecosystem's manifest; `check` validates the whole tree to IR; then
`import` pulls the Go deps with history and `vendor` pins the Rust/JS snapshots.
Finally `export` ships `sdk/go/client/` out to a local stand-in destination
(`acme/sdk-go`) as a branch + PR — the `client/` prefix stripped so the SDK sits
at the destination root, and `transforms` scrub the source on the way out
(`@--internal only--` lines deleted, `@--OSS only--` lines uncommented — the
classic Copybara visibility markers). The destination is a local bare repo so the
demo stays hermetic; against a real GitHub remote, export opens the PR via the
`gh` CLI. See `examples/export/` for the export config on its own and
`scripts/smoke-export.sh` for the standalone hermetic export proof.

The three generators differ in how much they can resolve offline. A Go module
path *is* its GitHub slug and its version *is* a tag, so `gen-go` is pure (no
network). Cargo and npm name packages by registry identity, which doesn't encode
the upstream repo — so `gen-cargo`/`gen-npm` look up the recorded repository URL
(crates.io / npm) and resolve the locked version's tag to a commit via
`git ls-remote`. Pin exact versions in `Cargo.lock` / `package-lock.json`; the
manifest requirement (`anyhow = "1"`) is only a fallback and may not match a tag.

## Imported `SRC` files become `ORIG_SRC`

`SRC` is CapyFun's package-marker filename. If an upstream dependency contains its
own file named `SRC`, importing it would otherwise be mistaken for a CapyFun
package. So every imported/vendored `SRC` file (at any depth) is renamed to
`ORIG_SRC` as it lands; CapyFun's own `SRC` at the destination is preserved.

## Notes

- All three `third_party/` trees are generated from their manifests:
  `third_party/go` by `gen-go`, `third_party/rust` by `gen-cargo`,
  `third_party/js` by `gen-npm`. Re-running the generators is idempotent.
- The generators vendor crates/packages from their **GitHub** source (a
  `git_repository` snapshot), not from registry tarballs — so the result is real,
  diffable source pinned to an upstream commit. A package whose registry records
  no GitHub repository is reported as skipped.
- Manifest lockfiles (`go.sum`, `Cargo.lock`, `package-lock.json`) here are a
  minimal hand-written subset; run the native tool (`go mod tidy`, `cargo
  generate-lockfile`, `npm install`) for real ones.
